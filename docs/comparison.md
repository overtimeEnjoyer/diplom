# Порівняння архітектурних підходів і технологій

## 1. Архітектурні підходи

| Критерій | Моноліт (VPS) | Чистий BaaS | Microservices | **Гібрид serverless/BaaS (обрано)** |
|----------|---------------|-------------|---------------|-------------------------------------|
| Швидкість розробки | Середня | Висока | Низька | **Висока** |
| Масштабованість | Обмежена без DevOps | Висока | Дуже висока | **Висока (auto-scale Functions)** |
| Вартість підтримки | Висока (сервер 24/7) | Низька–середня | Дуже висока | **Низька** |
| Vendor lock-in | Низький | Високий | Середній | **Середній–низький** |
| Контроль бізнес-логіки | Повний | Обмежений | Повний | **Повний** |
| Складність DevOps | Середня | Низька | Дуже висока | **Низька** |
| Реляційні дані | Відмінно | Складніше | Відмінно | **Відмінно (PostgreSQL)** |
| Тестованість | Висока | Середня | Складна | **Висока (services)** |
| Освітня платформа | Підходить | Підходить для MVP | Overkill | **Оптимально** |
| Перспектива масштабування | Вертикальна + репліки | Залежить від вендора | Горизонтальна | **Горизонтальна API + BaaS DB** |

**Висновок:** для дипломної роботи гібрид поєднує швидкий деплой, контроль над оплатами/доступами та реляційну модель контенту.

## 2. Express.js vs NestJS vs Fastify

| Критерій | **Express.js** | NestJS | Fastify |
|----------|----------------|--------|---------|
| Крива навчання | Низька | Висока | Середня |
| Екосистема | Максимальна | Велика | Зростає |
| Vercel/serverless | Відмінно | Можливо | Відмінно |
| Структура для диплому | Гнучка (layers) | Opinionated | Гнучка |
| Overhead | Мінімальний | Вищий | Низький |

**Обрано Express.js:** мінімалістичний, зрілий REST-стек, легко переноситься на Vercel, достатній для демонстрації шарів без зайвої складності Nest.

## 3. Sequelize vs Prisma vs Drizzle

| Критерій | **Sequelize** | Prisma | Drizzle |
|----------|---------------|--------|---------|
| Migrations | Так | Так | Так |
| Associations | Зрілі | Relations | Manual |
| PostgreSQL | Так | Так | Так |
| Read replicas | Підтримка | Обмежена | Обмежена |
| Досвід / література | Багато | Сучасний | Новий |

**Обрано Sequelize:** migrations, transactions, associations — критично для доступів і зв’язків методик.

## 4. PostgreSQL vs MongoDB vs Firestore

| Критерій | **PostgreSQL** | MongoDB | Firestore |
|----------|----------------|---------|-----------|
| Транзакції доступу | ACID | Обмежені | Обмежені |
| Зв’язки section→method | Природні | Embedding | Складні запити |
| JSONB для blocks | Так | Нативно | Документи |
| BaaS (Neon/Supabase) | Так | Atlas | Firebase |

**Обрано PostgreSQL:** структуровані освітні дані, цілісність оплат і доступів.

## 5. Vercel Functions vs AWS Lambda vs VPS

| Критерій | **Vercel Functions** | AWS Lambda | VPS |
|----------|----------------------|------------|-----|
| DevOps | Мінімальний | Середній | Високий |
| Cold start | Є | Є | Немає |
| Вартість на старті | Низька | Низька | Фіксована |
| Інтеграція з Git | Відмінна | Через CI | Ручна |

**Обрано Vercel Functions:** швидкий деплой API для дипломного прототипу; БД винесена в BaaS (не в функції).

## 6. BaaS DB vs власний API-шар

| Аспект | Тільки BaaS (Supabase client) | **BaaS DB + Express API** |
|--------|------------------------------|---------------------------|
| Складні правила оплати | Важко | **У services** |
| Payment confirmation | Provider webhook vs internal confirm | **Повний контроль доменної логіки** |
| Тести бізнес-логіки | Складніше | **Jest на services** |
| Переносимість | Прив’язка до SDK | **Стандартний REST** |

**Обрано комбінацію:** PostgreSQL як BaaS + власний API — баланс між швидкістю інфраструктури та контролем домену.
