# Звіт для куратора щодо виконаної дипломної роботи

## Тема та загальна мета роботи

У межах дипломної роботи я розробив серверну частину вебплатформи психологічної та освітньої підтримки, побудовану на базі **Strapi 5**. Основною метою було створення повноцінного backend-рішення, яке поєднує функції CMS, API для фронтенду, модулі автентифікації користувачів, керування платним доступом до контенту та інтеграцію з платіжною системою.

Результатом роботи став готовий backend-проєкт, який забезпечує:

- централізоване зберігання та керування психологічними методиками;
- реєстрацію та авторизацію користувачів;
- відновлення пароля через email-коди;
- оплату окремих розділів, тарифів і додаткового контенту;
- автоматичне надання доступів після успішної оплати;
- API для інтеграції з фронтендом;
- підготовку середовища для локального запуску та деплойменту.

## Технологічний стек

Під час реалізації роботи було використано такі технології та інструменти:

- **Node.js** як серверне середовище виконання;
- **TypeScript** для типізованої реалізації логіки;
- **Strapi 5** як headless CMS і серверна платформа;
- **PostgreSQL** та **SQLite** як варіанти бази даних для різних середовищ;
- **WayForPay** для приймання оплат;
- **SendGrid / Nodemailer** для email-відправлень;
- **Docker** для контейнеризації та підготовки до розгортання.

## Архітектура реалізованого backend-рішення

Проєкт побудований як модульний backend із чітким розділенням на конфігурацію, API-модулі, контент-типи та розширення Strapi. Архітектура охоплює такі частини:

- `config/` - конфігурація сервера, бази даних, middleware та email-плагінів;
- `src/api/` - бізнес-модулі прикладної логіки;
- `src/extensions/users-permissions/` - розширення стандартної моделі користувача Strapi;
- `src/components/` - повторно використовувані контентні компоненти;
- `docs/` - технічна документація для інтеграції та запуску;
- `Dockerfile` - контейнеризація проєкту для розгортання.

Таким чином, backend не обмежується лише CMS-функціональністю, а виконує роль повноцінного прикладного сервера для комерційного цифрового продукту.

## Що саме було реалізовано

### 1. Побудовано серверну частину на базі Strapi

Я створив серверний застосунок на основі Strapi 5 з використанням TypeScript. Було налаштовано структуру проєкту, конфігурацію запуску, збирання, старту в режимі розробки та продакшену. Також підготовлено контейнеризацію через Docker, що дозволяє розгортати систему на віддаленому сервері або хмарній платформі.

### 2. Реалізовано власну модель предметної області

У проєкті мною було описано та налаштовано кілька основних сутностей:

- **method-section** - розділ або категорія методик;
- **method** - окрема методика з текстовим і структурованим вмістом;
- **user-method-section** - зв’язок між користувачем і придбаним розділом;
- розширену модель **user**, до якої додано бізнес-поля доступу.

Для сутності користувача я додав додаткові атрибути:

- `makCardsAccess` - доступ до МАК-карток;
- `isMedium` - активований тариф Medium;
- `isPremium` - активований тариф Premium;
- `makFavoriteCardIds` - список улюблених МАК-карток;
- `userMethodSections` - список придбаних користувачем розділів методик.

Це дозволило адаптувати стандартну модель Strapi під прикладні потреби дипломного проєкту.

### 3. Реалізовано систему реєстрації та автентифікації

У backend я реалізував повний базовий цикл роботи з акаунтом користувача:

- реєстрацію нового користувача;
- логін через стандартний Strapi JWT-механізм;
- отримання даних поточного користувача;
- оновлення профілю;
- відновлення пароля через одноразовий email-код;
- підтримку додаткового коду підтвердження email для сумісності.

Особливістю реалізації стало те, що після реєстрації користувач одразу отримує JWT і може працювати в системі без додаткових ручних дій адміністратора. Для скидання пароля реалізовано окремий сценарій з 6-значним кодом, який має обмежений час дії.

### 4. Реалізовано email-взаємодію

У межах дипломної роботи я налаштував модуль відправлення електронної пошти з підтримкою двох варіантів:

- **SendGrid API** як основний канал;
- **Nodemailer/SMTP** як резервний варіант.

Через цей механізм система вміє:

- надсилати код для скидання пароля;
- надсилати код для підтвердження email у сумісному сценарії;
- обробляти форму зворотного зв’язку від користувачів.

Така реалізація підвищує надійність системи та забезпечує гнучкість під час розгортання в різних середовищах.

### 5. Реалізовано API для фронтенду

Одним із головних результатів моєї роботи стало формування повноцінного REST API, з яким може працювати клієнтська частина застосунку. Я підготував не лише самі ендпоінти, а й окрему документацію для інтеграції фронтенду.

Основні групи API:

- `auth` - реєстрація, логін, профіль, скидання пароля;
- `feedback` - надсилання форми зворотного зв’язку;
- `method-sections` - отримання розділів методик;
- `methods` - отримання окремих методик;
- `user-method-sections` - робота з придбаними користувачем розділами;
- `tariffs` - активація платних тарифів;
- `payments` - callback від платіжної системи і перевірка статусу оплати;
- `mak-cards` - робота з доступом і вподобаними МАК-картками.

Таким чином, backend став повноцінною інтеграційною основою для окремого frontend-застосунку.

### 6. Побудовано систему керування контентом методик

Через Strapi я організував збереження та структурування методичного контенту. Кожна методика містить не лише назву та slug, а й розширені поля:

- автора або джерело;
- підхід;
- цільову аудиторію;
- мету;
- час виконання;
- матеріали;
- основний зміст у форматі `blocks`;
- терапевтичний ефект;
- інструкцію;
- інтерпретацію;
- завершення;
- рефлексивні запитання.

За рахунок цього система дозволяє зберігати не просто окремі записи, а повноцінний навчально-психологічний контент зі структурованим наповненням.

### 7. Реалізовано персоналізований доступ до контенту

У системі реалізовано механізм, за яким користувач після оплати отримує доступ саме до тих матеріалів, які він придбав. Для цього я створив окрему сутність `user-method-section`, яка зв’язує користувача з конкретним розділом методик і містить ознаку оплаченого доступу.

Також реалізовано логіку:

- отримання списку розділів, доступних поточному користувачу;
- автоматичного відображення куплених розділів у профілі;
- синхронізації доступів після оновлення стану користувача.

Це дало змогу персоналізувати контент і забезпечити контроль доступу на рівні бізнес-логіки.

### 8. Реалізовано платіжну підсистему

Однією з найбільш важливих і технічно складних частин дипломної роботи стала інтеграція з **WayForPay**. Я реалізував повний цикл роботи з оплатою:

- створення платіжного посилання;
- генерацію унікального `orderReference`;
- перевірку callback від платіжної системи;
- валідацію підпису та ключових параметрів платежу;
- визначення типу оплати за `orderReference`;
- активацію відповідного доступу після підтвердження оплати;
- окремий ендпоінт для перевірки статусу платежу.

Система підтримує кілька типів платного доступу:

- оплата окремого розділу методик;
- тариф **Medium**;
- тариф **Premium**;
- доступ до **МАК-карток**.

Після успішної оплати backend автоматично змінює стан користувача та/або створює потрібні записи доступу, що усуває необхідність ручного адміністрування.

### 9. Реалізовано тарифну модель доступу

Я реалізував окрему логіку для двох тарифів:

- **Medium**;
- **Premium**.

Для кожного тарифу backend:

- ініціалізує оплату;
- повертає фронтенду `paymentUrl`;
- після callback активує відповідні прапорці доступу в користувача;
- синхронізує доступ до контенту після зміни стану облікового запису.

Тариф **Premium** додатково пов’язаний із розширеним доступом, зокрема до МАК-карток.

### 10. Реалізовано доступ до МАК-карток і список обраних

Окремим модулем я реалізував функціональність МАК-карток:

- ініціалізацію оплати доступу до карток;
- збереження прапорця доступу в профілі користувача;
- отримання списку улюблених карток;
- повну заміну списку улюблених;
- додавання або видалення окремої картки зі списку.

Для коректної роботи збереження я реалізував додаткову логіку синхронізації й захисту від конфліктів паралельних запитів.

### 11. Реалізовано системні lifecycle-механізми

У проєкті я використав lifecycle-підхід Strapi, щоб автоматично реагувати на створення та оновлення користувача. Після зміни полів доступу система:

- перевіряє стан тарифів;
- синхронізує доступ до методик;
- у разі потреби відкриває або відкликає доступ до контенту;
- очищає некоректні записи доступу.

Завдяки цьому логіка доступів працює не тільки через callback платежів, а й залишається узгодженою на рівні всієї системи.

### 12. Налаштовано базу даних для різних середовищ

У дипломному проєкті було передбачено роботу як у локальному середовищі, так і у віддаленому середовищі розгортання. Для цього я реалізував:

- режим роботи з **SQLite** для локальної та швидкої розробки;
- режим роботи з **PostgreSQL** для стабільного продакшен-середовища;
- підтримку SSL-налаштувань для розгортання на Render;
- параметризацію конфігурації через змінні оточення.

Це спрощує перенесення системи між середовищами та робить рішення придатним до практичного використання.

### 13. Підготовлено документацію для інтеграції та запуску

Окремим результатом роботи стала внутрішня технічна документація, у якій описано:

- як frontend має працювати з backend API;
- як підключати базу даних;
- які змінні оточення потрібно задати;
- як запускати систему локально;
- як працюють ключові сценарії автентифікації та оплати.

Документація потрібна для того, щоб систему могли підтримувати, тестувати та розвивати інші учасники команди.

## Практична цінність виконаної роботи

Практична цінність дипломної роботи полягає в тому, що створений backend є не демонстраційним макетом, а реальним прикладним рішенням для цифрового сервісу. Він поєднує CMS-підхід, комерційні сценарії доступу до контенту та інтеграцію з зовнішніми сервісами.

Розроблене рішення дозволяє:

- керувати психологічним контентом через адміністративну панель;
- надавати користувачам персоналізований доступ до матеріалів;
- здійснювати монетизацію через тарифи та окремі покупки;
- інтегрувати систему з клієнтським вебінтерфейсом;
- використовувати проєкт як основу для подальшого масштабування.

## Підсумок виконаної дипломної роботи

У ході виконання дипломної роботи я спроєктував і реалізував backend-платформу для психологічного сервісу з платним доступом до контенту. У результаті було створено модульну серверну систему, яка охоплює:

- CMS для керування методиками;
- систему користувачів і JWT-автентифікацію;
- email-сценарії;
- персоналізований доступ до контенту;
- інтеграцію з WayForPay;
- тарифікацію та монетизацію;
- API для взаємодії з фронтендом;
- підготовку до локального запуску та продакшен-розгортання.

Отже, поставлену мету дипломної роботи досягнуто: створено функціональний backend, який вирішує реальні прикладні завдання та може використовуватися як основа для повноцінного цифрового продукту.
